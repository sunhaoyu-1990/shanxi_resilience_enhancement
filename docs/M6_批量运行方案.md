# M6 OD-Section-Path 批量运行方案

## 一、背景与目标

### 1.1 处理规模

| 项目 | 数据 |
|------|------|
| Hive 表数量 | 至少 12 张（`gstx_exit_with_min_fee202601` ~ `gstx_exit_with_min_fee202612`） |
| 单表记录数 | 约 3,900 万条 |
| 总记录数 | 约 4.7 亿条 |
| 本地内存 | 2 GB |

### 1.2 目标

- **断点续跑**：中途崩溃可从上次位置恢复，已处理数据不丢失
- **批次实时落库**：每批处理完立即写入数据库，不在内存累积
- **增量更新**：后续新数据追加，已存在数据累加，不覆盖
- **rank 最终算**：所有批次/表跑完后统一计算排名

---

## 二、拓扑版本管理

### 2.1 规则

- 拓扑版本 = 选用 <= 数据月份的最新拓扑版本
- 例如：`gstx_exit_with_min_fee202603`（2026年3月数据）→ 用拓扑 `202512`
- 同一 `(enid, exid, numpath)` 可能因拓扑版本不同而映射出不同的 `fixed_ig`

### 2.2 表结构扩展

为支持多拓扑版本对比，map 表和 freq 表新增 `topo_version` 字段：

```sql
-- map 表新增字段
ALTER TABLE dwd_od_section_path_map
    ADD COLUMN topo_version VARCHAR(6) DEFAULT '202512';

-- freq 表新增字段
ALTER TABLE dwd_od_section_path_numpath_freq
    ADD COLUMN topo_version VARCHAR(6) DEFAULT '202512';
```

查询时按 `topo_version` 筛选，不同拓扑版本的数据**不合并**（因为是用不同拓扑算出来的，结果不可比）。

## 三、检查点（Checkpoint）机制

### 3.1 表结构

```sql
-- ============================================================
-- M6: 批量运行检查点表 DDL
-- ============================================================
-- 作用: 记录每个版本的处理进度，支持断点续跑
-- 主键: (table_name, version_yyyyMM)
-- ============================================================

CREATE TABLE IF NOT EXISTS m6_checkpoint (
    -- 表标识
    table_name       VARCHAR(64)  NOT NULL,
    version_yyyyMM    VARCHAR(6)  NOT NULL,

    -- 处理进度
    batch_offset      BIGINT      NOT NULL DEFAULT 0,   -- 已完成的最大偏移量
    records_processed BIGINT      NOT NULL DEFAULT 0,  -- 已处理记录数
    last_batch_time   VARCHAR(10) DEFAULT NULL,         -- 最后一批完成时间(HH:MM:SS)

    -- 状态
    status            VARCHAR(20) NOT NULL DEFAULT 'running',
                        -- running: 运行中
                        -- completed: 已完成
                        -- failed: 失败

    -- 元数据
    topo_version      VARCHAR(6)  NOT NULL DEFAULT '202512',  -- 该版本使用的拓扑版本
    created_at        TIMESTAMP   DEFAULT CURRENT_TIMESTAMP,
    updated_at        TIMESTAMP   DEFAULT CURRENT_TIMESTAMP,

    -- 主键（每个版本独立记录）
    CONSTRAINT pk_m6_checkpoint PRIMARY KEY (table_name, version_yyyyMM)
);

-- 索引
CREATE INDEX IF NOT EXISTS idx_m6cp_status ON m6_checkpoint(status);
CREATE INDEX IF NOT EXISTS idx_m6cp_version ON m6_checkpoint(version_yyyyMM);
```

### 3.2 初始数据

### 3.3 更新逻辑

```sql
-- 每批处理完成后更新 checkpoint
UPDATE m6_checkpoint SET
    batch_offset = :next_offset,                    -- 下一批次起始偏移量
    records_processed = records_processed + :batch_size,  -- 累计处理数
    last_batch_time = to_char(NOW(), 'HH24:MI:SS'),
    updated_at = CURRENT_TIMESTAMP
WHERE table_name = :table_name
  AND version_yyyyMM = :version;
```

## 四、数据写入策略

### 4.1 内存管理

- 每批处理完立即 upsert 到数据库，**不在内存累积**
- 内存只保留当前批次数据：约 **1~5 MB**
- 多进程并行在批次内，不影响内存

```
批次数：39,225,614 / 100,000 ≈ 393 批/表
每批内存：~1-5 MB
总内存占用恒定：~5 MB
```

### 4.2 freq 表：每批累加写入

```sql
-- 每批完成后执行（累加，不覆盖）
INSERT INTO dwd_od_section_path_numpath_freq (
    enid, exid, numpath, fixed_intervalgroup,
    version_yyyyMM, topo_version,
    ig_count, source_flag
) VALUES (...)
ON CONFLICT (enid, exid, numpath, fixed_intervalgroup, version_yyyyMM)
DO UPDATE SET
    ig_count = dwd_od_section_path_numpath_freq.ig_count + EXCLUDED.ig_count,
    -- rank 暂不更新，留到最后统一算
    updated_at = CURRENT_TIMESTAMP;
```

**幂等保证**：相同数据重复处理 → count 翻倍 → 最终 rank 不变

### 4.3 map 表：从 freq 表派生

**不每批更新 map 表**，而是等所有批次跑完后，从 freq 表统一派生：

```sql
-- 所有批次/表跑完后执行一次
WITH agg AS (
    SELECT
        enid, exid, numpath, version_yyyyMM,
        SUM(ig_count) AS total_trip_cnt,
        MAX(topo_version) AS topo_version
    FROM dwd_od_section_path_numpath_freq
    WHERE version_yyyyMM = :version
    GROUP BY enid, exid, numpath, version_yyyyMM
),
best AS (
    SELECT
        f.enid, f.exid, f.numpath, f.version_yyyyMM,
        f.fixed_intervalgroup,
        f.ig_count,
        a.topo_version,
        ROW_NUMBER() OVER (
            PARTITION BY f.enid, f.exid, f.numpath, f.version_yyyyMM
            ORDER BY f.ig_count DESC, f.fixed_intervalgroup
        ) AS rn
    FROM dwd_od_section_path_numpath_freq f
    JOIN agg a USING (enid, exid, numpath, version_yyyyMM)
    WHERE f.version_yyyyMM = :version
)
INSERT INTO dwd_od_section_path_map (
    enid, exid, numpath, version_yyyyMM,
    fixed_intervalpath, intervalpath_cnt, total_trip_cnt,
    path_freq_ratio, topo_version, source_flag
)
SELECT
    enid, exid, numpath, version_yyyyMM,
    fixed_intervalgroup,
    ig_count,
    (SELECT total_trip_cnt FROM agg WHERE
        agg.enid = best.enid AND agg.exid = best.exid
        AND agg.numpath = best.numpath AND agg.version_yyyyMM = best.version_yyyyMM),
    ROUND(ig_count::NUMERIC /
        (SELECT total_trip_cnt FROM agg WHERE
            agg.enid = best.enid AND agg.exid = best.exid
            AND agg.numpath = best.numpath AND agg.version_yyyyMM = best.version_yyyyMM), 4),
    topo_version,
    'hive_computed'
FROM best WHERE rn = 1
ON CONFLICT (enid, exid, numpath, version_yyyyMM)
DO UPDATE SET
    fixed_intervalpath = EXCLUDED.fixed_intervalpath,
    intervalpath_cnt   = EXCLUDED.intervalpath_cnt,
    total_trip_cnt     = EXCLUDED.total_trip_cnt,
    path_freq_ratio    = EXCLUDED.path_freq_ratio,
    topo_version       = EXCLUDED.topo_version,
    updated_at         = CURRENT_TIMESTAMP;
```

### 4.4 rank 最终计算

```sql
-- 所有表跑完后执行一次
UPDATE dwd_od_section_path_numpath_freq AS t
SET    ig_rank = ranked.r_rank,
       updated_at = CURRENT_TIMESTAMP
FROM   (
    SELECT id, topo_version,
           RANK() OVER (
               PARTITION BY enid, exid, numpath, version_yyyyMM, topo_version
               ORDER BY ig_count DESC
           ) AS r_rank
    FROM dwd_od_section_path_numpath_freq
) AS ranked
WHERE  t.id = ranked.id
   AND t.topo_version = ranked.topo_version;
```

## 五、断点续跑逻辑

### 5.1 启动流程

```
启动
  ↓
读取所有 status='running' 的 checkpoint（按 version_yyyyMM 升序）
  ↓
遍历每个版本：
  ↓
从 checkpoint.batch_offset 继续读 Hive 批次
  ↓
每批处理 → upsert → 更新 checkpoint.batch_offset
  ↓
所有批次完成 → status='completed' → 处理下一版本
  ↓
所有版本完成 → 执行 rank 计算 SQL
```

### 5.2 故障恢复

| 故障点 | 损失 | 恢复行为 |
|--------|------|---------|
| 表X批次N崩溃 | 批次N的数据 | 重启 → 读到 checkpoint (表X, offset=N×batch_size) → 从批次N继续 |
| 手动 Ctrl+C | 同上 | 重启 → 自动从断点继续 |
| 数据库崩溃 | freq表和checkpoint完整 | 重启 → 读取checkpoint → 从中断处继续 |

### 5.3 幂等保证

- 相同批次数据重复处理 → freq 表 count 翻倍
- 最终 map 表和 rank 不受影响（以累加后的 count 为准）
- `ON CONFLICT DO UPDATE SET ig_count = ig_count + EXCLUDED.ig_count`

## 六、拓扑版本自动选择

### 6.1 选择规则

根据数据月份，选择 <= 数据月份的最新拓扑版本：

```python
TOPO_VERSIONS = ['202312', '202409', '202411', '202507', '202512']

def select_topo_version(data_version: str) -> str:
    """
    选择 <= data_version 的最新拓扑版本

    例如：
        data_version='202603' → '202512'  (202512 < 202603，取最接近的)
        data_version='202512' → '202512'  (相等，直接取)
        data_version='202410' → '202409'  (202512 > 202410，取次新)
    """
    available = [v for v in TOPO_VERSIONS if v <= data_version]
    return max(available) if available else TOPO_VERSIONS[0]
```

### 6.2 批次内拓扑版本固定

每个 checkpoint 记录其使用的 `topo_version`，后续追加数据时：

```python
# 从 checkpoint 读取 topo_version
cp = repo.get_checkpoint(table_name, version)
topo_version = cp['topo_version']

# 所有批次使用相同拓扑版本（确保一致性）
```
