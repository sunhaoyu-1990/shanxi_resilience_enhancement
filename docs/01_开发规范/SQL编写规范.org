#+TITLE: SQL 编写规范
#+STARTUP: overview
#+OPTIONS: toc:2 num:t

* 1. 适用范围
本文档适用于本项目所有正式 SQL 文件，包括：
- sql/ddl/
- sql/dml/
- sql/views/
- sql/checks/

* 2. 总体原则

** 2.1 SQL 是核心分析逻辑载体
本项目中：
- 统计
- 聚合
- 汇总
- 规则匹配
- 落表
优先通过 SQL 完成。

Python 主要做编排，不建议把 SQL 逻辑强行改写成 Python 数据处理。

** 2.2 可读性优先于花哨写法
SQL 必须让人能看懂：
- 输入表是什么
- 输出表是什么
- 粒度是什么
- 关键逻辑是什么

** 2.3 可复现
每个正式 SQL 都必须：
- 可独立审查
- 可重复执行
- 输入输出明确
- 参数来源清楚

** 3.1 DDL
存放路径：

* 4. 文件命名规范
文件名必须表达：
- 动作
- 目标表
- 业务含义

示例：
- build_dim_section_info.sql
- build_dws_section_capacity_day.sql
- check_dws_impacted_od_flow_day.sql
- select_best_diversion_path.sql

避免：
- temp.sql
- test1.sql
- run_final_new2.sql

* 5. SQL 头部注释规范
每个正式 SQL 文件顶部必须有注释块，说明：

#+begin_example
-- 功能：构建施工期间收费单元日可通行当量表
-- 模块：M1
-- 输入表：dim_section_info, dwd_scheme_section_map, dim_capacity_rule
-- 输出表：dws_section_capacity_day
-- 粒度：section_id × stat_date × vehicle_type
-- 关键字段：section_id, stat_date, vehicle_type, capacity_pcu, scheme_id
#+end_example

没有头部说明的 SQL 不允许进入正式目录。

* 6. 书写规范

** 6.1 关键字大小写
建议 SQL 关键字统一大写：
- SELECT
- FROM
- WHERE
- JOIN
- GROUP BY
- ORDER BY

表名和字段名保持小写下划线。

** 6.2 字段逐行展开
SELECT 字段较多时，建议逐行展开：

#+begin_src sql
SELECT
    section_id,
    stat_date,
    vehicle_type,
    capacity_pcu,
    scheme_id
FROM dws_section_capacity_day;
#+end_src

** 6.3 JOIN 显式写法
禁止使用隐式逗号连接。
统一使用显式 JOIN：

#+begin_src sql
SELECT
    a.section_id,
    b.scheme_id
FROM dim_section_info AS a
JOIN dwd_scheme_section_map AS b
    ON a.section_id = b.section_id;
#+end_src

** 6.4 CTE 优先
复杂 SQL 优先拆成 WITH 结构，保证分段可读：

#+begin_src sql
WITH base_flow AS (
    ...
),
capacity_base AS (
    ...
),
impact_calc AS (
    ...
)
SELECT ...
FROM impact_calc;
#+end_src

* 7. 粒度说明规范
每个 SQL 必须明确当前结果粒度。

尤其是以下表必须写清粒度：
- 收费单元级
- section_number 归并级
- OD 级
- OD-Path 级
- 方案级
- 日期级
- 车型级

如果粒度不清楚，后续结果极易错位。

* 8. 参数规范

** 8.1 参数占位
建议通过统一参数模板注入，不在 SQL 中写死方案 ID、日期范围等。

例如：
- {{ scheme_id }}
- {{ stat_date_start }}
- {{ stat_date_end }}

具体由 Python 层渲染。

** 8.2 禁止写死环境差异
禁止在 SQL 中写死：
- 库名差异
- 路径差异
- 测试环境限定条件

* 9. 落表规范

** 9.1 落表前必须明确
- 是全量重建
- 还是增量写入
- 还是按分区覆盖

** 9.2 建议写明主键粒度
例如：
- section_id + stat_date + vehicle_type
- scheme_id + od_id + path_id

** 9.3 created_at / updated_at
正式结果表建议补齐：
- created_at
- updated_at

** 9.4 source_flag
涉及规则补全、插补、经验修正的表，建议增加：
- source_flag

用于标识：
- actual
- filled
- rule
- api

* 10. 校验 SQL 规范
每个核心结果表至少有一份配套校验 SQL，建议覆盖：

- 主键重复检查
- 非空检查
- 数值非负检查
- 关键关联存在性检查
- 汇总一致性检查

示例：
#+begin_src sql
SELECT
    section_id,
    stat_date,
    vehicle_type,
    COUNT(*) AS cnt
FROM dws_section_capacity_day
GROUP BY
    section_id,
    stat_date,
    vehicle_type
HAVING COUNT(*) > 1;
#+end_src

* 11. section_number 特别规范
涉及收费单元归并、路径归并、断面映射时，若使用 section_number，必须明确写出：

- 是否以收费单元粒度输出
- 是否以 section_number 粒度先归并再分配
- 归并逻辑是否可追溯到 section_id

不得模糊混用 section_id 与 section_number。

* 12. 性能与可维护性原则
当前阶段优先级是：
1. 逻辑正确
2. 结果可解释
3. 便于联调
4. 再考虑性能优化

不要过早为了性能写成难以维护的复杂 SQL。

* 13. 禁止事项
- 禁止无头部说明的正式 SQL
- 禁止隐式 JOIN
- 禁止把复杂业务逻辑全部塞进一个巨型 SELECT 不拆层
- 禁止结果粒度不说明
- 禁止写死方案 ID / 日期等业务参数
- 禁止输出字段名含义不清
