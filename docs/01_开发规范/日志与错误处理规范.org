#+TITLE: 日志与错误处理规范
#+STARTUP: overview
#+OPTIONS: toc:2 num:t

* 1. 目标
本文档用于统一本项目的日志输出与错误处理方式，保证：
- 问题能定位
- 任务能追踪
- 失败能重跑
- 结果可审计

* 2. 总体原则

** 2.1 日志服务于排障与追踪
日志不是为了“看起来很多”，而是为了回答：
- 哪个任务在跑
- 跑到哪一步
- 用了哪些输入
- 产出了哪些输出
- 花了多少时间
- 为什么失败

** 2.2 错误必须可解释
错误不能只有一行 “failed”。
必须尽量说明：
- 模块
- 步骤
- 关键参数
- 关键表
- 原始异常

** 2.3 可恢复与不可恢复要区分
不是所有错误都要直接终止整个链路。
需要区分：
- 可恢复错误
- 可跳过错误
- 致命错误

* 3. 日志规范

** 3.1 日志级别
- DEBUG：调试细节，仅开发环境开启
- INFO：正常任务节点
- WARNING：异常但可继续
- ERROR：当前步骤失败
- CRITICAL：系统级严重故障

** 3.2 日志最小字段
每条关键日志建议包含：
- timestamp
- level
- module_code
- task_id
- scheme_id
- step_name
- message

关键业务节点还建议补充：
- input_tables
- output_tables
- affected_rows
- duration_seconds
- batch_no
- request_id（若有接口调用）

** 3.3 关键节点必须打日志
每个任务至少在以下节点打印 INFO 日志：
- 任务开始
- 配置加载完成
- 输入校验完成
- SQL 开始执行
- SQL 执行完成
- 输出校验完成
- 任务结束

* 4. 任务日志要求

** 4.1 任务开始日志
至少包含：
- module_code
- task_id
- scheme_id
- run_params
- operator（若有）

** 4.2 SQL 执行日志
至少包含：
- sql_name
- input_tables
- output_table
- start_time
- end_time
- affected_rows

** 4.3 任务结束日志
至少包含：
- status
- duration_seconds
- output_tables
- summary_message

* 5. 错误分类规范
建议统一错误分类：

** 5.1 ConfigError
配置类错误，例如：
- yaml 缺字段
- 环境变量缺失
- 配置值非法

** 5.2 DatabaseError
数据库类错误，例如：
- 连接失败
- SQL 执行失败
- 事务提交失败

** 5.3 DataValidationError
数据校验类错误，例如：
- 主键重复
- 非空字段为空
- 关键关联缺失
- 数值非法

** 5.4 BusinessRuleError
业务规则类错误，例如：
- 规则表为空
- 最优路径规则未定义
- 施工方案与收费单元无法映射

** 5.5 ExternalServiceError
外部服务错误，例如：
- 路径接口失败
- 返回结果格式异常
- 超时

* 6. 错误处理原则

** 6.1 必须保留原始异常
不要只输出“执行失败”，必须保留原始异常摘要。

** 6.2 必须带上下文
错误日志至少应包含：
- module_code
- step_name
- task_id
- key_params
- input_table / output_table

** 6.3 禁止静默失败
禁止：
#+begin_src python
try:
    ...
except Exception:
    pass
#+end_src

** 6.4 可恢复错误处理
对于可恢复错误，可记录 WARNING 并继续，但必须满足：
- 已明确不会污染最终结果
- 已记录恢复策略
- 已在结果中留下标记

例如：
- 某条候选路径接口调用失败，退回到已有主路径集合

* 7. 重跑规范

** 7.1 可重跑要求
每个模块任务设计时应考虑：
- 是否支持 overwrite
- 是否支持按 scheme_id 重跑
- 是否支持按日期区间重跑
- 是否支持只重跑单模块

** 7.2 重跑前检查
重跑前至少明确：
- 覆盖哪些表
- 是否清理旧数据
- 是否保留历史批次

** 7.3 建议增加批次号
对关键结果表建议预留：
- batch_no
或
- version_no

便于追溯不同批次结果。

* 8. 接口错误返回规范
若后续提供查询服务，错误返回建议统一结构：

#+begin_src json
{
  "code": 3002,
  "message": "output table write failed",
  "data": {},
  "request_id": "REQ_20260403_001"
}
#+end_src

其中：
- code：错误码
- message：面向调用方的简要描述
- request_id：便于追踪日志

* 9. 当前阶段建议的失败策略

** 9.1 致命失败直接终止
以下情况建议直接 FAIL：
- 配置缺失
- 输入主表不存在
- 输出核心表创建失败
- 关键规则表为空
- 关键主键重复严重

** 9.2 可恢复失败允许降级
以下情况可考虑 WARNING + 降级：
- 某些非关键候选路径获取失败
- 某些非关键说明性字段缺失
- 某些扩展输出未生成

** 9.3 最终状态
任务最终状态建议统一为：
- SUCCESS
- FAILED
- PARTIAL_SUCCESS

* 10. 禁止事项
- 禁止 print 替代正式日志
- 禁止吞掉异常
- 禁止没有上下文的错误信息
- 禁止接口只返回“系统异常”而无 request_id
- 禁止任务失败后不记录失败步骤
