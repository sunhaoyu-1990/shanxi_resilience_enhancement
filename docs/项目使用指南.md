## 1. 项目是做什么的

**一句话**：当高速公路因改扩建施工而封闭车道时，用数据回答四个问题——

| # | 问题 | 通俗解释 |
|---|------|----------|
| 1 | 通行能力下降多少？ | 本来3车道，施工占1条，还剩多少运力？ |
| 2 | 多少车流受影响？ | 哪些路线的车会受影响？影响有多大？ |
| 3 | 司机们会怎么绕行？ | 有人绕路走、有人中途下高速，各有多少？ |
| 4 | 通行费收入变化多少？ | 绕路导致收费增加还是减少？总共多少？ |

**一个贯穿全文的例子**：假设 G65 西安-铜川段某处施工，占用1条车道。用项目工具可以查出：哪些起终点对（OD对）受影响、流量减少多少、哪些车中途下站或绕行、通行费损失多少。

**项目全称**：陕交控多路段改扩建韧性提升项目（一期）

**一期范围**：以统计方法、经验规则和工程化实现为主，不做动态寻优或交通仿真。

## 2. 必须先理解的领域概念

> 这是本指南最重要的章节。如果这些概念没搞清楚，后面的操作和代码都会看不懂。

### 2.1 收费单元 (section_id)

**是什么**：高速公路上两个相邻门架/收费站之间的最小计费路段。

**类比**：地铁的"区间"（A站到B站是一段），收费单元就是高速的"区间"。

**命名**：16位编码，如 `G007061003000210`

**关键属性**：里程(length)、设计时速、车道数、方向(上行/下行)、roadtype(路段类型)

### 2.2 串行路段编号 (section_number)

**是什么**：在同一个串行路径上的多个收费单元共享同一个编号。

**为什么需要**：同一条高速公路上，不同位置的门架切分出多个收费单元，但它们属于同一条"路"。用 section_number 归并后，不同车走同一路径的不同门架记录可以被归到一起。

**类比**：同一条公交线路上的不同站点，线路编号相同。

**例子**：section_number=2 下可能包含 G065W61001000110、G065W61001000210 等多个收费单元

> **避坑**：section_id 和 section_number 是两个不同的东西！section_id 是唯一的收费单元标识（16位），section_number 是串行路段编号（整数）。一个 section_number 下可以有多个 section_id。

### 2.3 OD对 (Origin-Destination)

**是什么**：一辆车的入口站(enid)到出口站(exid)的一次完整出行。

**类比**：地铁从A站进B站出。

**命名**：enid/exid 是14位收费站编码，如 `G0005610010060`

**用途**：OD对是流量统计的基本维度。我们关心"从哪到哪"有多少车。

### 2.4 收费单元路径 (intervalgroup / numpath)

**intervalgroup**：一辆车实际经过的收费单元ID序列，用 `|` 分隔。

例子：`G007061003000210|G007061003000310|G007061003000410`

**numpath**：将 intervalgroup 中的 section_id 替换为 section_number 后去重得到的简短序列。

去重步骤：相邻去重 → 配对去重 → 相邻去重

例子：`2|35|325`

**为什么需要 numpath**：不同车走同一条路，intervalgroup 可能略有差异（门架检测差异），但 numpath 一致，可以归并。

### 2.5 拓扑修复 (intervalgroup fix)

**问题**：原始数据中 intervalgroup 经常有"断裂"——两相邻门架之间路网不直接连通。

**修复方法**：用 pgRouting 最短路径算法补全缺失的中间节点。

**代码位置**：`src/modules/m2_od_flow/interval_fixer.py` — `fix_intervalgroup()`

**四种修复情况**：
- Case 1：拓扑连通，直接保留
- Case 2：拓扑断裂，用最短路径补全
- Case 3：方向反转（上下行搞反，10↔20互换）
- Case 4：尾部断裂，跳过

### 2.6 版本管理 (version_yyyymm)

**为什么需要版本**：收费单元、收费站、路网拓扑会随时间变化。

**格式**：YYYYMM（如 202603 表示2026年3月版本）

**版本选择逻辑**：找 `effect_date <= 目标日期` 的最新版本。

**三类版本数据**：

| 数据类型 | 明细表 | 版本配置表 |
|----------|--------|-----------|
| 收费单元路径 | dwd_section_path | dim_section_path_version |
| 收费站信息 | dwd_toll_station | dim_toll_station_version |
| 路网拓扑 | dwd_tom_noderelation | dim_tom_noderelation_version |

### 2.7 数据来源标识 (source_flag)

**是什么**：让每个数字可以追溯来源，下游模块可据此判断数据置信度。

| 值 | 含义 |
|----|------|
| `actual` | 真实采集数据 |
| `filled` | 统计补全数据 |
| `rule` | 规则生成数据 |
| `api` | 外部接口数据 |
| `computed` | 计算派生数据 |

### 2.8 交控集团与还贷性

**判断逻辑**：收费单元ID前11位匹配 `dim_toll_road.收费路段编号`，若 `路段性质='还贷性'` 则属于交控集团。

**为什么重要**：项目只关注交控集团（陕西交控）管辖路段的通行费影响。

### 2.9 通行费计算

**核心公式**：`单元通行费 = length(米)/1000 × feebykm(元/公里)`

**费率来源**：`dim_road_vehicle_fee_map`，按 roadtype × feetype × vehicle_type 查询

**比较逻辑**：同时计算最短路径和实际路径通行费，取较小值（因为司机按最短路径收费）

**代码位置**：`src/common/toll_calculator.py` — `calculate_toll_fee()`

## 3. 数据仓库分层架构

**类比理解**：

| 层级 | 类比 | 说明 |
|------|------|------|
| ODS | 原材料仓库 | 原始数据直接入库，最小处理 |
| DIM | 零件库 | 维表（不变/缓慢变化），如收费单元、收费站 |
| DWD | 组装车间 | 明细事实表（原子事实），如路径、拓扑 |
| DWS | 半成品仓库 | 轻度汇总事实表，如流量统计 |
| ADS | 成品展厅 | 应用结果表，可直接用于汇报 |

**数据流向**：

```
原始数据源                    数据仓库分层                     应用输出
┌──────────┐              ┌─────────────────────────┐      ┌──────────┐
│ CSV通行流水│──→ ODS ──→ │ DIM 维表                 │      │          │
│ Excel基础表│──→ DIM ──→ │ dwd_section_path         │──→   │ M3 影响分析│
│ Hive表     │──→ DWD ──→ │ dwd_od_section_path_map  │──→   │ M4 分流优化│
│            │            │ dwd_tom_noderelation     │      │ M5 费用测算│
│            │            ├──────────────────────────┤      │          │
│            │            │ DWS 汇总事实              │──→   │ CSV报告   │
│            │            │ dws_section_od_path_flow │      │ 查询API   │
│            │            ├──────────────────────────┤      └──────────┘
│            │            │ ADS 应用结果              │
│            │            │ ads_toll_impact_result   │
└──────────┘              └─────────────────────────┘
```

**依赖规则**：上层依赖下层，不允许反向依赖。ODS → DIM/DWD → DWS → ADS → APP

## 9. M3 交通影响分析 — 最核心的模块

### 9.1 三个子流程的关系

```
流程1: 受影响OD-Path流量查询
    "哪些路线受了影响？影响多大？"
        │
        ├──→ 流程3: 绕行记录检测
        │       "谁绕路了？绕了多少？"
        │
        └──→ 流程2: 中途下站检测
                "谁半路下车了？下了多少？"
        │
        └──→ 综合汇总
                "每个OD对的总影响全貌"
```

**执行顺序**：1 → 3 → 2（流程1先划定范围，流程3和2分别检测两种司机行为）

### 9.2 流程1：受影响OD-Path流量查询

**通俗解释**：想象小区门口修路。你首先要知道：哪些人的通勤路线必须经过这条路？这些人每天有多少？

**核心数据结构**：倒排索引 `dwd_section_path_bridge` — 从施工段快速定位经过它的OD路径

**三级递进过滤**：
1. 过滤①：受影响路径单条流量 ≤ `min-affected-path-flow` 去掉
2. 过滤②：没有受影响路径的OD对整个去掉
3. 过滤③：OD对总流量 ≤ `min-flow` 去掉

```bash
uv run python -m src.jobs.run_affected_od_query \
  --section-ids "G007061003000210|G300161001002220" \
  --start-date 20260315 --end-date 20260415
```

### 9.3 流程3：绕行记录检测

**通俗解释**：小区修路，平时张三从南门进、东门出。施工后你发现张三从南门进、西门出了——他换了终点。

**核心算法**："找同端、判异端"
- O离施工段近 → "找D判O"（终点没变但起点换了）
- D离施工段近 → "找O判D"（起点没变但终点换了）

```bash
uv run python -m src.jobs.run_detour_record_detector \
  --affected-od-csv analysis_results/affected_od_flow.csv \
  --section-ids "G007061001000610" \
  --start-date 20260301 --end-date 20260331
```

### 9.4 流程2：中途下站检测

**通俗解释**：你坐公交从A到D，中间B站修路不通了，你被迫在B站下车，走几步到旁边的C站重新上车。这个"中途下车→再上车"的过程就是中途下站。

**核心算法**：滑动窗口逐天匹配
- 每天只保留18:00后下站的车作为"待匹配"
- 第二天把前一天待匹配车和当天记录做配对
- 内存恒定约100MB，不随搜索天数增长

```bash

  --affected-od-csv analysis_results/affected_od_flow.csv \
  --start-date 20260315 --end-date 20260415
```

## 13. 核心算法与公共模块

> 本节面向想要阅读或修改代码的学生，说明核心算法的位置和原理。

### 13.1 最短路径检索方法

所有路径检索函数位于 `sql/pgrouting/` 目录：

| 函数 | 文件 | 算法 | 用途 |
|------|------|------|------|
| `find_shortest_path_pgr(start, end, version)` | `query_functions_pgr.sql` | Dijkstra | 单条最短路径 |
| `find_k_shortest_paths_pgr(start, end, version, k)` | `query_functions_pgr.sql` | K-Shortest Paths | 前K条最短路径 |
| `find_shortest_path_excluding(start, end, version, exclude_nodes)` | `query_functions_excluding.sql` | Dijkstra+排除 | 排除指定节点的最短路径 |
| `find_top_n_paths_pgr(start, end, version, n)` | `query_top_n_paths_pgr.sql` | 贪心法 | Top-N不同路径 |

**调用关系**：

```
TollCalculator.calculate_toll_fee()
    └── get_shortest_path() ──→ find_shortest_path_pgr()

TopologyChecker.shortest_path()
    └── fix_intervalgroup() ──→ find_shortest_path_pgr()

M3 detour_record_service  ──→ find_shortest_path_pgr()  (绕行验证)
M3 mid_trip_exit_service  ──→ find_shortest_path_pgr()  (中途下站验证)
```

### 13.2 路径补全方法 (intervalgroup fix)

**代码位置**：`src/modules/m2_od_flow/interval_fixer.py`

**核心函数**：`fix_intervalgroup(intervalgroup, topology, ...)` — 返回 `IntervalFixResult`

**算法**：滑动窗口三元组 (X1, X2, X3)

| Case | 条件 | 处理 |
|------|------|------|
| 1 | X1→X2 相邻 且 X2→X3 相邻 | 正常连通 |
| 2 | X1→X2 断裂 且 X2→X3 相邻 | 最短路径补全X1→X2 |
| 3 | X1→X2 断裂 且 X2→X3 断裂 | 尝试方向反转(10↔20)，否则最短路径补全 |
| 4 | X1→X2 相邻 且 X2→X3 断裂 | 滑动窗口前移 |

**TopologyChecker**：缓存 `dwd_tom_noderelation` 邻接关系，调用 `find_shortest_path_pgr` 做最短路径补全。

**调用者**：M2 (`flow_stat_service.py`)、M6 (`service.py`)

### 13.3 通行费计算

**代码位置**：`src/common/toll_calculator.py`

**核心函数**：`calculate_toll_fee(enid, exid, intervalgroup, vehicle_type, version)`

**逻辑**：
1. 如果 enid/exid 非空，查询最短路径
2. 如果 intervalgroup 也非空，计算两条路径的通行费，取较小值
3. 单元通行费 = `length(米)/1000 × feebykm(元/公里)`
4. roadtype 3/4 跳过费用但计入里程
5. 交控单元（`dim_toll_road.路段性质='还贷性'`）单独累加

**详见**：`docs/业务逻辑记录.md` BL-003

### 13.4 路段级OD匹配 (SectionOdMatcher)

**代码位置**：`src/common/section_od_matcher.py`

**核心函数**：`match_section_od(enid, exid, dataDate)` — 返回排序后的 section_od 字符串（如 `"35|325"`）

**三级解析链路**：
1. station_map：14位站ID → section_number
2. sectionMap：16位收费单元ID → section_number
3. 下钻两层：通过 `dwd_tom_noderelation` 查后继门架再查映射

**详见**：`docs/业务逻辑记录.md` BL-009

### 13.5 版本解析 (version_utils)

**代码位置**：`src/common/version_utils.py`

**核心函数**：`get_nearest_version(data_date, version_table)` — 返回最近的版本号

**逻辑**：查询版本表中 `version_yyyymm <= 目标日期` 的最新记录，默认返回 `"202603"`。

**详见**：`docs/业务逻辑记录.md` BL-010

### 13.6 代码架构模式

每个模块统一结构：

```
m{N}_xxx/
├── service.py       # 业务编排层：调用repository，记录日志，异常捕获
├── repository.py    # 数据访问层：调用SqlRunner执行SQL
├── schema.py        # Pydantic模型：TaskParams、TaskResult等
└── checks.py        # 数据质量校验
```

**数据流向**：Job入口 → Service编排 → Repository数据访问 → SqlRunner执行SQL

**SqlRunner** (`src/common/sql_runner.py`)：
- `load_sql()` — 加载 `sql/dml/` 下的SQL文件
- `render_sql()` — Jinja2模板渲染参数
- `execute_sql()` / `fetch_all()` / `fetch_one()` — 执行SQL

### 14.3 常用查询模板

```sql
-- 查某个收费单元属于哪个路段、是否交控
SELECT sp.id, sp.roadid, tr.收费路段名称, tr.路段性质
FROM dwd_section_path sp
LEFT JOIN dim_toll_road tr ON LEFT(sp.roadid, 11) = tr.收费路段编号
WHERE sp.id = 'G007061003000210' AND sp.version_yyyymm = '202603';

-- 查某个OD对有哪些路径
SELECT enid, exid, numpath, fixed_intervalpath
FROM dwd_od_section_path_map
WHERE enid = 'G0005610010060' AND exid = 'G000561001001820';

-- 查某个收费单元的流量趋势
SELECT stat_hour, SUM(flow_cnt) AS hourly_flow
FROM dws_section_od_path_flow_hour_202603
WHERE section_id = 'G007061003000210'
GROUP BY stat_hour
ORDER BY stat_hour;

-- 查施工段影响了哪些OD
SELECT DISTINCT enid, exid, numpath
FROM dwd_section_path_bridge b
JOIN dwd_od_section_path_map m ON b.od_section_path_id = m.id
WHERE b.section_id = 'G007061001000610';
```

## 17. 常见问题与避坑指南

### 环境相关

| 问题 | 解决方法 |
|------|----------|
| 数据库连不上 | 检查 `.env` 配置、防火墙、VPN |
| Hive连不上 | 确认内网访问权限 |
| 内存不足 | 减小 `--batch-size` 或 `--workers` |

### 数据相关

| 问题 | 原因 | 解决方法 |
|------|------|----------|
| 修复失败记录过多 | topo-version 与数据版本不匹配 | 传入正确的 `--topo-version` |
| OD查询结果为空 | section-ids 或 version 不对 | 检查 section-ids 是否正确、version 是否匹配 |
| 日表找不到 | 日表带日期后缀 | 日表名如 `dws_section_od_path_flow_hour_20260301`，不是统一月表 |

### 运行相关

| 问题 | 原因 |
|------|------|
| M0/M1/M4/M5 运行后只输出占位信息 | 正常，这些模块业务逻辑尚未填充 |
| 流水线中某个模块失败 | 用 `--continue-on-error` 跳过 |
| 想重跑某个月 | 先删除对应日表，再重新运行 |

### 概念相关（最容易混淆的点）

| 易混淆概念 | 区别 |
|------------|------|
| section_id vs section_number | section_id 是16位唯一收费单元标识；section_number 是整数串行路段编号，一个编号下可有多个section_id |
| intervalgroup vs numpath | intervalgroup 是section_id序列（长，如`A\|B\|C`）；numpath 是section_number去重序列（短，如`2\|35`） |
| 日表 vs 月表 | 日表名带日期后缀 `_20260301`，月表名带月份后缀 `_202603`；查询时优先使用日表 |
| 2025同期对比 | 将施工期日期映射到2025年同期（如2026-03-15 → 2025-03-15），用于对比 |
