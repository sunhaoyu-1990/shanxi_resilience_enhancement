* 1. 项目背景
本项目面向多路段改扩建及养护工程并发施工场景，目标是建立一套可工程化落地的后端算法处理链路，用于支撑施工期间的通行能力评估、交通影响分析、分流路径生成和通行费影响测算。

当前阶段为一期最小可用版本（MVP）建设阶段。

* 2. 一期目标
一期目标不是一次性完成完整研究闭环，而是优先实现一条可运行、可交付、可查询的后端算法主链路。
当前阶段以统计分析、经验规则和工程化实现为主。

一期重点输出包括：
- 数据底座与标准化数据表
- 收费单元施工映射结果
- 施工期间收费单元可通行当量
- 受影响OD与迁移流量结果
- 替代路径和分流管控单元
- 通行费增收/减收结果
- 最优绕行方案

* 3. 一期主链路
当前统一采用以下模块划分：

** M0 数据工程
负责原始数据接入、施工方案入库、收费单元施工映射、OD-Path 映射底座建设。

** M1 通行能力评估
根据施工方案与车道数配置，生成施工期间收费单元日可通行当量。

** M2 流量与OD迁移统计补全
负责收费单元/OD/Path 日流量统计、路径归并、缺失补全。

** M3 交通影响分析
根据通行能力与历史流量关系，对施工影响进行自然量修正，输出受影响OD与迁移流量结果。

** M4 分流路径优化
负责候选路径获取、施工影响路径匹配、分流管控点生成、路径里程与费用计算。

** M5 通行费影响测算
根据通行里程/金额增幅与绕行比例规则，生成最优绕行方案并输出通行费影响结果。

* 4. 非当前阶段重点内容
以下内容当前阶段不作为一期硬性交付主链：
- 完整动态寻优体系
- 分流效果评估闭环
- 通行费影响动态优化
- 路网级多工程协同深度决策
- 完整交通组织优化研究
- 大规模自动报告闭环

相关能力后续可在现有工程骨架上逐步增强。

* 5. 技术路线
** 5.1 总体技术路线
- Python + SQL
- PostgreSQL + PostGIS
- 离线批处理为主
- 查询服务为辅
- YAML 配置
- Org 文档

** 5.2 工程组织原则
- SQL 负责核心统计与分析逻辑
- Python 负责参数、编排、日志、校验、服务封装
- 工程代码与研究代码分离
- 表设计先于算法实现
- 规范先于编码

* 6. 数据底座关键约束
** 6.1 section_number
收费单元维表必须包含 =section_number= 字段。
该字段表示“同一条单一路径上的所有收费单元归属同一编号”，是路径合并、收费单元归并、断面映射的重要基础字段。

** 6.2 路网拓扑
当前路网拓扑基础形态为：
- 本收费单元ID
- 下一个收费单元ID
- 一对多关系

后续可在此基础上扩展更多拓扑和空间属性。
