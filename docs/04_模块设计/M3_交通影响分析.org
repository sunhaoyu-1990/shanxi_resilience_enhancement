* 1. 模块定位

M3 负责分析施工对交通流的影响，核心回答"哪些路线受影响、影响多大、司机如何应对"。

M3 当前包含三个已实现的子流程：
- **流程1**：受影响OD-Path流量查询 — 划定"受灾范围"
- **流程3**：绕行记录检测 — 找到"绕道走的人"
- **流程2**：中途下站检测 — 找到"被迫中途下车的人"

以及一个综合汇总服务，将三个流程的结果合并为每个OD对的总影响全貌。

* 2. 三个子流程关系

执行顺序：1 → 3 → 2（流程1先划定范围，流程3和2分别检测两种司机行为）

```
流程1: 受影响OD-Path流量查询
    "哪些路线受了影响？影响多大？"
        │
        ├──→ 流程3: 绕行记录检测
        │       "谁绕路了？绕了多少？"
        │
        └──→ 流程2: 中途下站检测
                "谁半路下车了？下了多少？"
        │
        └──→ 综合汇总 (ImpactSummaryService)
                "每个OD对的总影响全貌"
```

* 3. 流程1：受影响OD-Path流量查询

详见 [[../业务逻辑记录.md::BL-006][BL-006: 受影响OD-Path查询逻辑]]

**核心数据结构**：倒排索引 dwd_section_path_bridge
- 从施工段ID快速定位经过该段的所有 OD-Path 映射记录

**分步逻辑**：
1. 用倒排索引查找经过施工段的路径
2. 提取OD对，反查全部路径
3. 标记受影响/未受影响
4. 查询施工期和2025同期流量
5. 三级递进过滤（单条路径→OD对→总流量）
6. 批量计算通行费
7. 输出CSV

**代码位置**：src/modules/m3_impact_analysis/affected_od_service.py

* 4. 流程3：绕行记录检测

详见 [[../业务逻辑记录.md::BL-007][BL-007: 绕行记录检测逻辑]]

**核心算法**："找同端、判异端"
- O离施工段近 → "找D判O"（终点没变但起点换了）
- D离施工段近 → "找O判D"（起点没变但终点换了）

**4项检查条件**：
1. 记录的终点站在对应集合里
2. 记录的起点 ≠ 原始OD的起点O
3. 记录起点→O的最短路径经过施工段
4. 最短路径节点数 < max-construction-sections + 施工段数

**代码位置**：src/modules/m3_impact_analysis/detour_record_service.py

* 5. 流程2：中途下站检测

详见 [[../业务逻辑记录.md::BL-008][BL-008: 中途下站检测逻辑]]

**核心算法**：滑动窗口逐天匹配
- 每天只保留18:00后下站的车作为"待匹配"
- 第二天配对前一天的待匹配车和当天记录

**5项检查条件**：
1. 第1程的起点 = OD的起点O
2. 第2程的终点 = OD的终点D
3. 0 < 重新上车时间 - 中途下车时间 ≤ 6小时
4. 中途下车站 ≠ 重新上车站
5. 中途下车站→重新上车站的最短路径经过施工段

**代码位置**：src/modules/m3_impact_analysis/mid_trip_exit_service.py

* 6. 综合汇总

ImpactSummaryService 将三个流程的结果合并为一张综合汇总表，按OD对+车型给出：
- 流量变化（施工期 vs 2025同期）
- 绕行损失
- 中途下站损失

**代码位置**：src/modules/m3_impact_analysis/impact_summary_service.py

| 来源 | 说明 |

M3 的 service.py 中还有基于 dws_impacted_od_flow_day 的基础流程（供需对比、影响比例计算），
该部分为框架占位，业务逻辑尚未填充。三个子流程是独立实现的专项分析工具。
