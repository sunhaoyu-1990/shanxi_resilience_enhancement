* 1. 模块定位

M2 是一期主链中承上启下的关键模块，负责从原始通行流水数据统计收费单元-OD(path)级别的小时流量。

M2 的核心职责：
- 从 CSV 通行流水或 Hive 表读取原始通行记录
- 对 intervalgroup 进行拓扑修复（补全断裂、修正方向）
- 将收费单元ID序列映射为 section_number 并去重生成 numpath
- 按 od_section_path_id 查找或插入映射记录
- 按小时粒度聚合流量，写入 PostgreSQL

* 2. 实际实现（非原始设计）

实际实现使用 `dws_section_od_path_flow_hour` 系列表（月表或日表），而非原始设计中的 `dws_section_od_flow_day`。

有两个实现版本：
1. **m2_flow_od_completion/** — 框架占位版，通过 SQL 驱动的 service/repository 模式
2. **m2_od_flow/** — 实际运行版，基于 CSV/Hive 直接读取 + Python 内存聚合

本文档描述 m2_od_flow（实际运行版）。

** 3.1 输入

| 来源 | 说明 |
|------|------|
| CSV通行流水 | 每月~3900万行，16GB，格式：gstx_exit_with_min_fee{YYYYMM}.csv |

| 表名 | 说明 | 主键粒度 |
|------|------|---------|
| dws_section_od_path_flow_hour_{YYYYMM} | 月表版本 | (section_id, od_section_path_id, stat_hour, vehicle_type) |

* 4. 核心算法

** 4.1 数据处理流程

CSV读取 → intervalgroup修复 → numpath映射 → od_section_path_id查找/插入 → 小时截断+去重 → 内存聚合 → 批量upsert

** 4.2 intervalgroup 拓扑修复

详见 [[../业务逻辑记录.md::BL-004][BL-004: intervalgroup 拓扑修复算法]]

核心：滑动窗口三元组(X1,X2,X3)，4种Case处理断裂和方向错误，通过 pgRouting 最短路径补全缺失节点。

** 4.3 numpath 去重

详见 [[../业务逻辑记录.md::BL-005][BL-005: numpath 去重算法]]

步骤：section_id→section_number替换 → 相邻去重 → 配对去重 → 再次相邻去重

** 4.4 od_section_path_id 查找/插入

对每条修复后的记录，用 (enid, exid, numpath, version) 在 dwd_od_section_path_map 中查找 od_section_path_id。未找到时自动 INSERT 新记录并获取新 ID。

** 4.5 小时截断与去重

- 入口时间（entime）截断到小时作为 stat_hour
- 同一 tradeid 只计算一次（按 tradeid 去重）

** 4.6 内存聚合与批量 upsert

- 使用 defaultdict 在内存中聚合 (section_id, od_section_path_id, stat_hour, vehicle_type) → flow_cnt
- 每 mini_batch（默认50000条）刷盘一次
- 使用 ON CONFLICT ... DO UPDATE SET flow_cnt = table.flow_cnt + EXCLUDED.flow_cnt 累加

* 5. 并行处理

M2 支持多进程并行处理：
- 使用 multiprocessing.Pool (fork 模式)
- 预扫描 CSV 建立行偏移索引，Worker seek 到指定位置
- 共享数据（sectionMap、TopologyChecker）通过 fork CoW 共享
- 每个 Worker 创建独立的 Repository 和数据库连接
- 支持断点续跑（checkpoint 机制）
