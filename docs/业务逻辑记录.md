### BL-001: 判断收费单元是否属于交控集团

**编号**: BL-001

**名称**: 判断收费单元是否属于交控集团

**创建日期**: 2026-04-13

**业务描述**:
根据收费单元 ID 判断该收费单元是否属于交控集团。

---

### 详细逻辑

#### 输入
- `section_id` - 收费单元 ID（字符串）
- 参考数据文件：`research/data/收费路段.xls`

#### 处理步骤

1. **提取收费单元 ID 前 11 位**
   - 从 `section_id` 中提取前 11 位字符
   - 例如：`section_id = "ABC123456789XYZ"` → 提取 `"ABC1234567"`

2. **匹配收费路段编号**
   - 使用提取的 11 位字符
   - 在 `research/data/收费路段.xls` 文件中查找
   - 匹配字段：`收费路段编号`

3. **判断路段性质**
   - 找到匹配的记录后，查看 `路段性质` 字段
   - 如果 `路段性质` = `"还贷性"` → **属于交控集团**
   - 其他情况（包括未匹配到）→ **不属于交控集团**

#### 输出
- `is_shanjiao` - 布尔值
  - `true` - 属于交控集团
  - `false` - 不属于交控集团

---

### 参考数据说明

**文件位置**: `research/data/收费路段.xls`

**关键字段**:
| 字段名 | 说明 | 示例 |
|--------|------|------|
| `收费路段编号` | 收费路段编号（用于匹配） | `"ABC1234567"` |
| `路段性质` | 路段性质（用于判断） | `"还贷性"` |

---

### 使用场景

此逻辑在以下模块中使用：

| 模块 | 用途 |
|------|------|
| **M0 数据工程** | 收费单元属性标记 |
| **M1 通行能力评估** | 仅计算交控集团收费单元 |
| **M2 流量与 OD 补全** | 根据业务需求筛选 |
| **其他模块** | 根据业务需求筛选 |

---

### 伪代码示例

```python
def is_shanjiao_section(section_id: str, toll_segments_df: DataFrame) -> bool:
    """
    判断收费单元是否属于交控集团

    Args:
        section_id: 收费单元 ID
        toll_segments_df: 收费路段数据 DataFrame

    Returns:
        bool: True=交控, False=非交控
    """
    # 1. 提取前 11 位
    if len(section_id) < 11:
        return False

    segment_prefix = section_id[:11]

    # 2. 匹配收费路段编号
    matched = toll_segments_df[
        toll_segments_df["收费路段编号"] == segment_prefix
    ]

    if matched.empty:
        return False

    # 3. 判断路段性质
    segment_type = matched.iloc[0]["路段性质"]

    return segment_type == "还贷性"
```

```sql
    SELECT
        section_id,
        LEFT(section_id, 11) AS segment_prefix
    FROM dwd_section_info
),
matched_segment AS (
    SELECT
        sp.section_id,
        ts.路段性质
    FROM section_prefix sp
    LEFT JOIN dim_toll_segment ts
        ON sp.segment_prefix = ts.收费路段编号
)
SELECT
    section_id,
    CASE
        WHEN 路段性质 = '还贷性' THEN true
        ELSE false
    END AS is_shanjiao
FROM matched_segment;
```

### 版本选择逻辑

#### 输入
- `data_date` - 数据日期（YYYY-MM-DD 格式）

#### 处理逻辑

1. **提取年月**
   - 从 `data_date` 中提取年月，格式为 `YYYYMM`
   - 例如：`data_date = "2024-05-15"` → `"202405"`

2. **查找适用版本**
   - 找到所有版本文件的生效年月 ≤ 目标年月
   - 选择其中最大的（最新的）版本
   - 例如：目标年月 `"202405"`，适用版本是 `"202401"`

3. **确定文件路径**
   - 根据选定的版本年月，构造文件名
   - 例如：版本 `"202401"` → 文件 `"202401单元唯一路径.xlsx"`

#### 输出
- `version_yyyyMM` - 适用版本年月（YYYYMM 格式）
- `file_path` - 适用版本文件的完整路径

---

### 伪代码示例

```python
from pathlib import Path
from datetime import datetime
from typing import Tuple

# 版本配置
VERSION_CONFIG = {
    "202401": "2024-01-01",
    "202409": "2024-09-01",
    "202412": "2024-12-01",
    "202507": "2025-07-01",
    "202512": "2025-12-01",
    "202603": "2026-03-01",
}

DATA_DIR = Path("research/data/基础数据/2024-2026年收费单元唯一路径")


def get_section_path_version(data_date: str) -> Tuple[str, Path]:
    """
    根据数据日期获取适用的收费单元唯一路径版本

    Args:
        data_date: 数据日期，格式 YYYY-MM-DD

    Returns:
        (version_yyyyMM, file_path): 版本年月和文件路径
    """
    # 1. 解析日期
    dt = datetime.strptime(data_date, "%Y-%m-%d")
    target_yyyymm = dt.strftime("%Y%m")

    # 2. 找到所有生效年月 <= 目标年月的版本
    applicable_versions = [
        v for v in VERSION_CONFIG.keys()
        if v <= target_yyyymm
    ]

    if not applicable_versions:
        raise ValueError(f"没有找到适用于日期 {data_date} 的版本")

    # 3. 选择最新的适用版本
    selected_version = max(applicable_versions)

    # 4. 构造文件路径
    file_name = f"{selected_version}单元唯一路径.xlsx"
    file_path = DATA_DIR / file_name

    if not file_path.exists():
        raise FileNotFoundError(f"版本文件不存在: {file_path}")

    return selected_version, file_path
```

## BL-003: 通行费计算

**编号**: BL-003

**名称**: 通行费计算

**创建日期**: 2026-05-09

**模块位置**: `src/common/toll_calculator.py`

**业务描述**:
根据入口/出口收费站和收费单元列表，计算路径通行费、交控通行费、总里程和交控里程。支持最短路径查询和 intervalgroup 直接计算两种模式。

---

### 函数接口

```python
from src.common.toll_calculator import calculate_toll_fee, TollFeeResult

def calculate_toll_fee(
    enid: str,              # 入口收费站14位编码（可为空）
    exid: str,              # 出口收费站14位编码（可为空）

    vehicle_type: int,     # 车型（1-4客车, 11-26货车）
    version: str = "202512" # 路网版本
) -> TollFeeResult:
    """计算通行费，返回通行费、交控通行费、里程、交控里程"""
```

### 逻辑说明

#### 入口判断逻辑

```
IF enid 为空 OR exid 为空:
    # 直接计算 intervalgroup
    直接计算 intervalgroup 的通行费、里程、交控里程
    path_type = "direct"
ELSE:
    # enid/exid 都不为空，查最短路径
    shortest_path = find_shortest_path_pgr(enid, exid, version)

    IF intervalgroup 为空:
        # 直接使用最短路径
        计算 shortest_path 的通行费、里程、交控里程
        path_type = "shortest"
    ELSE:
        # 计算两者，比较取较小值
        shortest_fee = calculate_path_fee(shortest_path)
        intervalgroup_fee = calculate_path_fee(split_intervalgroup(intervalgroup))

        IF shortest_fee <= intervalgroup_fee:
            返回 shortest_fee 结果
        ELSE:
            返回 intervalgroup_fee 结果
        path_type = "intervalgroup"
```

#### 路径通行费计算（calculate_path_fee）

对给定收费单元列表，遍历计算：

1. **查询收费单元信息**（`dwd_section_path`）
   - `length`: 里程（米）
   - `roadtype`: 路段类型（1-普通公路, 2-桥隧加收, 3-西铜高速, 4-一二级路）
   - `feeKtype`: 客车费率类型
   - `feeHtype`: 货车费率类型
   - `roadid`: 所属路段编号

2. **里程累加**
   - 所有收费单元的 `length` 累加到 `total_length_meters`

3. **roadtype 检查**
   - roadtype 1/2: 正常计算通行费
   - roadtype 3/4: 跳过（不计入费用，但计入里程），记录到 `skipped_sections`

4. **费率选择**
   - 客车（vehicle_type 1-4）: 使用 `feeKtype`
   - 货车（vehicle_type 11-26）: 使用 `feeHtype`

5. **费率查询**（`dim_road_vehicle_fee_map`）
   - 条件: `roadtype`, `feetype`, `vehicle_type`
   - 返回: `feebykm`（每公里收费金额，元/公里）

6. **单元金额计算**
   ```
   section_fee = length(米) / 1000 * feebykm(元/公里)
   ```

7. **交控判断**（`dim_toll_road`）
   - `roadid[:11]` = `收费路段编号`
   - 如果 `路段性质` = '还贷性'，则为交控单元
   - 交控单元的金额和里程分别累加到 `control_fee_yuan` 和 `control_length_meters`

---

### 表关联关系

| 源字段 | 目标表 | 目标字段 | 说明 |
|--------|--------|----------|------|
| `section_id` | `dwd_section_path` | `id` | 收费单元编号 |
| `dwd_section_path.roadid[:11]` | `dim_toll_road` | `收费路段编号` | 路段关联 |
| `dim_toll_road.路段性质` | - | - | '还贷性'=交控 |
| `roadtype + feetype + vehicle_type` | `dim_road_vehicle_fee_map` | `feebykm` | 费率查询 |

---

### 费率类型映射

| 车型 | 费率来源 | 说明 |
|------|----------|------|
| 1-4（客车） | feeKtype | 1:甲类, 2:乙类, 3:丙类, 4:丁类 |
| 11-26（货车） | feeHtype | 1:甲类, 2:乙类, 3:丙类, 4:丁类 |

---

| roadtype | 含义 | 处理方式 |
|----------|------|----------|

| 2 | 桥隧加收 | 正常计算 |
| 3 | 西铜高速 | 跳过（计入里程，不计入费用） |
| 4 | 一二级路 | 跳过（计入里程，不计入费用） |

## BL-004: intervalgroup 拓扑修复算法

**编号**: BL-004

**名称**: intervalgroup 拓扑修复算法

**创建日期**: 2026-05-18

**模块位置**: `src/modules/m2_od_flow/interval_fixer.py`

**业务描述**:
对收费单元序列（intervalgroup）进行补全修复，处理两类问题：(1) 遗漏——相邻单元在拓扑上不相邻，中间有缺失节点；(2) 错误——中间单元方向识别错误（应为反向单元）。

---

### 函数接口

```python
from src.modules.m2_od_flow.interval_fixer import fix_intervalgroup, fix_intervalgroup_batch

def fix_intervalgroup(
    intervalgroup: str,       # 用 | 分隔的收费单元序列
    topology: TopologyChecker, # 拓扑检查器实例
    intervaltimegroup: str = "", # 对应的时间戳序列
    enid: str = "",           # 入口站ID（用于首部补全）
    exid: str = "",           # 出口站ID（用于尾部补全）
    entime: str = "",         # 入口时间
    extime: str = "",         # 出口时间
    max_iterations: int = 100 # 最大迭代次数
) -> IntervalFixResult:
    """修复 intervalgroup，返回修复结果"""

def fix_intervalgroup_batch(
    records: list[dict],          # 待修复记录列表
    topology: Optional[TopologyChecker], # 拓扑检查器
    version: str = "202512"       # 拓扑版本
) -> list[IntervalFixResult]:
    """批量修复，逐条调用 fix_intervalgroup"""
```

---

### 核心算法：滑动窗口三元组

对序列中的每个三元组 (X1, X2, X3)，判断 X1→X2 和 X2→X3 是否在拓扑上直接相邻：

| Case | 条件 | 处理 |
|------|------|------|
| Case 1 | X1→X2 相邻 且 X2→X3 相邻 | 正常连通，滑动窗口前移 |
| Case 2 | X1→X2 断裂 且 X2→X3 相邻 | 用 pgRouting 最短路径补全 X1→X2 之间的缺失节点 |
| Case 3 | X1→X2 断裂 且 X2→X3 断裂 | 尝试 X2 的反向单元（末2位 10↔20 互换），若反向后连通则替换；否则用最短路径补全 |
| Case 4 | X1→X2 相邻 且 X2→X3 断裂 | 滑动窗口前移，待下一轮判断 |

---

### 首尾补全

当传入 `enid`/`exid` 时，它们会临时插入序列首尾参与滑动窗口，修复完成后从输出中移除。这解决了入口站→首门架和末门架→出口站之间的断裂问题。

### 时间插值

修复过程中新增的节点，其时间戳在相邻已知时间点之间线性等间隔插值。

---

### TopologyChecker

```python
class TopologyChecker:
    def __init__(self, version: str = "202512")
    def topo_check(self, from_section: str, to_section: str) -> bool  # 判断直接相邻
    def shortest_path(self, start: str, end: str) -> Optional[list[str]]  # 调用 find_shortest_path_pgr
    def topo_next(self, section_id: str) -> set[str]  # 返回后继节点集合
```

- 预加载 `dwd_tom_noderelation` 的全部邻接关系到内存缓存
- `shortest_path` 调用 SQL 函数 `find_shortest_path_pgr`（Dijkstra 算法）
- fork-safe：子进程需调用 `_reset_pg_connection()` 重置连接

---

### 返回结构

```python
@dataclass
class IntervalFixResult:
    tradeid: str           # 交易ID
    original: str          # 原始 intervalgroup
    fixed: str             # 修复后 intervalgroup
    fixed_timegroup: str   # 修复后时间戳序列
    changes: list[IntervalFixChange]  # 修复变更记录
    error: Optional[str]   # 错误信息

@dataclass
class IntervalFixChange:
    position: int   # 在序列中的位置（0-based）
    original: str   # 原始单元ID
    fixed: str      # 修复后单元ID
    reason: str     # "reverse_fix" | "path_fill" | "reverse_path_fill"
```

---

### 使用场景

| 模块 | 用途 |
|------|------|
| **M2 流量统计** | 修复 CSV 通行记录的 intervalgroup，确保路径完整后再做流量聚合 |
| **M6 OD-Section-Path映射** | 修复 Hive 导出记录的 intervalgroup，确保映射准确 |

### 备注

- 方向反转（10↔20）是常见的修复原因，因为门架检测有时把上下行方向搞反
- `max_iterations=100` 防止超长路径导致无限循环

## BL-005: numpath 去重算法

**编号**: BL-005

**名称**: numpath 去重算法

**创建日期**: 2026-05-18

**模块位置**: `src/modules/m2_od_flow/interval_fixer.py`（内部逻辑，分散在 flow_stat_service.py 和 M6 service.py 中）

**业务描述**:
将 intervalgroup（收费单元ID序列）转换为 numpath（section_number 去重序列），用于归并不同车辆走同一路径但有微小门架差异的记录。

---

### 算法步骤

1. **将 section_id 替换为 section_number**：从 `dwd_section_path` 查询每个 section_id 对应的 section_number
2. **相邻去重**：如果相邻两个 section_number 相同，只保留一个（表示车辆在同一串行路段上经过多个门架）
3. **配对去重**：如果序列中出现 A|B|A 模式（即同一编号出现两次，中间夹一个不同编号），且中间的编号已出现在前方，则去重
4. **再次相邻去重**：配对去重后可能产生新的相邻重复

---

### 示例

```
intervalgroup: G065W61001000110|G065W61001000210|G065W61001000310|G300161001002220
→ section_number序列: 2|2|2|35
→ 相邻去重: 2|35
→ numpath: "2|35"
```

---

### 使用场景

| 模块 | 用途 |
|------|------|
| **M2 流量统计** | 将修复后的 intervalgroup 转为 numpath，用于 od_section_path_id 查找/插入 |
| **M6 OD-Section-Path映射** | 生成 numpath 频率表 `dwd_od_section_path_numpath_freq` |
| **M3 受影响OD查询** | numpath 是 OD-Path 映射的关键标识 |

## BL-007: 绕行记录检测逻辑

**编号**: BL-007

**名称**: 绕行记录检测逻辑

**创建日期**: 2026-05-18

**模块位置**: `src/modules/m3_impact_analysis/detour_record_service.py`

**业务描述**:
在收费流水中找出那些"本来该去A站但去了别的站"或"本来从B站出发却从别站出发"的车，通过路网最短路径验证改线确实是因为施工导致的，识别绕行记录并统计流量和通行费损失。

---

### 核心算法：找同端、判异端

1. **判断O和D谁离施工段近**：用 pgRouting 计算OD两端到施工段的最短路径节点数
2. **分类**：
   - O近 → "找D判O"：终点没变但起点换了（在D端找记录，判断O端是否改变）
   - D近 → "找O判D"：起点没变但终点换了（在O端找记录，判断D端是否改变）
3. **构建快速过滤集合**：只看起点/终点沾边的记录
4. **逐天扫描收费CSV流水**：对预过滤后的记录做绕行判定

---

### 绕行判定的4项检查（以"找D判O"为例）

| # | 条件 | 通俗解释 |
|---|------|----------|
| 1 | 记录的终点站在"找D判O"的D集合里 | 这辆车确实到了某个受影响OD的终点 |
| 2 | 记录的起点 ≠ 原始OD的起点O | 起点确实换了 |
| 3 | 记录起点→O的最短路径经过施工段 | 换起点确实和施工有关 |
| 4 | 最短路径节点数 < max-construction-sections + 施工段数 | 路径不会太离谱地长 |

---

### 关键参数

| 参数 | 默认值 | 含义 |
|------|--------|------|
| `--max-sections` | 5 | O/D到施工单元最短路径节点数上限，超过则不算"靠近" |
| `--max-construction-sections` | 6 | 最短路径中施工段个数上限，超过则不算绕行 |

---

## BL-008: 中途下站检测逻辑

**编号**: BL-008

**名称**: 中途下站检测逻辑

**创建日期**: 2026-05-18

**模块位置**: `src/modules/m3_impact_analysis/mid_trip_exit_service.py`

**业务描述**:
在收费流水中找出那些"上了高速、本来要去远方，结果中途被迫下站，过一会儿又重新上站继续走"的车辆，验证中途下站确实是因为前方施工导致的，统计中途下站的流量和通行费损失。

---

### 核心算法：滑动窗口逐天匹配

- 每天只保留**18:00后下站**的车作为"待匹配"对象（因为中途下车后重新上车通常在几小时内，跨天的可能性最大）
- 第二天把前一天的"待匹配"车和当天的记录做配对
- 内存优化：只保留一天的待匹配记录 + 当天全部记录，内存恒定约100MB

---

### 中途下站的5项检查

| # | 条件 | 通俗解释 |
|---|------|----------|
| 1 | 第1程的起点 = OD的起点O | 第一段行程确实是从出发点出发的 |
| 2 | 第2程的终点 = OD的终点D | 第二段行程确实到达了目的地 |
| 3 | 0 < 重新上车时间 - 中途下车时间 ≤ 6小时 | 下站和再上车之间隔了合理的时间 |
| 4 | 中途下车站 ≠ 重新上车站 | 下站=上站说明只是正常停车，不算被迫下站 |
| 5 | 中途下车站→重新上车站的最短路径经过施工段 | 下站确实是因为前方施工被逼的 |

---

### 内部常量

| 常量 | 值 | 含义 |
|------|-----|------|
| MAX_GAP | 6小时 | 中途下车到重新上车的最大时间间隔 |
| 傍晚截止 | 18:00 | 只保留18:00后下站的车做跨天匹配 |

### 版本选择逻辑

1. 将输入日期（YYYYMMDD 或 YYYYMM）截断为 YYYYMM
2. 在指定版本表中查询 `version_yyyymm <= yyyymm` 的最新记录
3. 未找到时返回默认值 `"202603"`

---

### 支持的版本表

| 版本表 | 对应数据 | 版本列表 |
|--------|----------|----------|
| `dim_tom_noderelation_version` | 路网拓扑结构 | 202401, 202409, 202412, 202507, 202512 |
| `dim_section_path_version` | 收费单元唯一路径 | 202401, 202409, 202412, 202507, 202512, 202603 |
