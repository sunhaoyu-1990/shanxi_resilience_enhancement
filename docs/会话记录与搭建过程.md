用户提出了一系列具体问题，我逐一解答：

#### 问题 2.1: sql\dml 是干什么的
**答案**: DML（Data Manipulation Language）目录存放数据加工 SQL，按模块拆分（m0~m5），用于从基础表加工生成业务结果表。

#### 问题 2.2: sql\views 是干什么的
**答案**: views 目录存放视图定义 SQL，用于封装常用查询逻辑，简化查询服务编写。

#### 问题 2.3: @src\app 下面放的都是干什么的
**答案**: app 目录存放应用核心基础设施：
- `enums.py` - 全局枚举（ModuleCode, TaskStatus, SourceFlag 等）
- `settings.py` - 配置管理（Pydantic BaseSettings）
- `db.py` - 数据库连接与会话管理
- `exceptions.py` - 统一异常体系
- `logger.py` - 日志配置

#### 问题 2.4: @src\common 那这个又是做什么的
**答案**: common 目录存放跨模块通用工具：
- `sql_runner.py` - SQL 加载、渲染、执行封装（核心！）
- `time_utils.py` - 日期解析、批次号生成
- `validators.py` - 输入校验
- `file_utils.py` - 文件操作

#### 问题 2.5: @src\jobs 是干什么的
**答案**: jobs 目录存放命令行任务入口，是离线批处理的统一入口点：
- `run_m0.py` ~ `run_m5.py` - 各模块任务入口
- `run_pipeline.py` - 全链路流水线入口

#### 问题 2.6: @src\modules 是干什么的
**答案**: modules 目录存放核心业务模块（M0~M5），每个模块按统一结构组织：
- `m{N}_xxx/`
  - `service.py` - 业务编排层
  - `repository.py` - 数据访问层
  - `schema.py` - Pydantic 模型
  - `checks.py` - 数据校验

#### 问题 2.7: @src\services 是干什么的
**答案**: services 目录存放查询服务，用于对外提供数据查询 API：
- `query_api/` - FastAPI 查询服务
  - `main.py` - FastAPI 应用入口
  - `schemas.py` - API 响应模型
  - `endpoints/` - API 端点

#### 问题 2.8: @src\tests 怎么使用，需要有哪些测试内容
**答案**: tests 目录存放测试代码，要求 80%+ 覆盖率：
- 单元测试 - 单个函数、工具
- 集成测试 - API 端点、数据库操作
- SQL 校验测试 - 数据质量校验

#### 问题 2.9: @research 和 @outputs 是干什么用的
**答案**:
- `research/` - 研究与试验代码（与工程分离，不进入生产）
- `outputs/` - 临时输出目录（不提交 Git）

### 第11步：执行安装遇到的问题及解决

#### 问题 11.1：Windows 换行符导致脚本无法执行

**错误信息**:
```
-bash: ./install_postgres.sh: /bin/bash^M: bad interpreter: No such file or directory
```

**解决方案**: 使用 dos2unix 转换文件格式

```bash
sudo apt install -y dos2unix
dos2unix install_postgres.sh
```

---

#### 问题 11.2：PostgreSQL 官方仓库 404 错误

**错误信息**:
```
Err:13 http://apt.postgresql.org/pub/repos/apt focal-pgdg Release
  404  Not Found
```

**解决方案**: 使用 PostgreSQL 归档仓库

```bash
echo "deb http://apt-archive.postgresql.org/pub/repos/apt/ focal-pgdg main" | sudo tee /etc/apt/sources.list.d/pgdg.list
```

---

#### 问题 11.3：postgresql.conf 配置语法错误

**错误信息**:
```
Error: invalid line 426 in /etc/postgresql/15/main/postgresql.conf: :efault_statistics_target = 100
```

**原因**: 配置文件中多了一个冒号

**解决方案**: 修复配置文件

```ini
# 错误的
:efault_statistics_target = 100

# 正确的
default_statistics_target = 100
```

#### 问题 12.3：pgAdmin 连接超时（VPN 问题）

**现象**:
- 服务器端 PostgreSQL 运行正常
- 从服务器本地可以连接
- Windows 能 ping 通服务器
- 但 pgAdmin 连接超时

**排查过程**:
1. 发现用户开了 VPN（Tailscale）
2. 断开 VPN 后 Windows IP 是 `192.168.16.135`
3. 服务器 IP 是 `192.168.0.75`
4. 两个设备不在同一个网段！

**原因**:
- 防火墙只允许了 `192.168.0.0/24` 网段
- 但 Windows 在 `192.168.16.0/24` 网段

**解决方案**:
```bash
# 允许整个 192.168.0.0/16 网段
sudo ufw allow from 192.168.0.0/16 to any port 5432
```

**结果**: pgAdmin 成功连接！

### 第23步：Top N 路径函数 Bug 修复

**问题描述**:
- 初始实现的 `find_top_n_paths_pgr` 只返回1条路径，无法找到多条替代路径

**问题分析**:
- SQL排除条件 `source = ALL(v_exclude_nodes) AND target = ALL(v_exclude_nodes)` 逻辑错误
- 该条件要求边的**两个端点同时被禁用**才能排除，实际几乎不满足

**解决方案**:
- 改为 `NOT (source = ANY(v_exclude_nodes) OR target = ANY(v_exclude_nodes))`
- 即边的**任意一端被禁用**就排除该边

**修复后测试结果**:
| 路径 | 里程 | 节点数 |
|------|------|--------|
| 1 | 54,136 米 | 11 |
| 2 | 123,582 米 | 16 |
| 3 | 1,000,169,450 米 | 18 |
| 4 | 4,000,000,000 米 | 4 |

- 总耗时: 81ms
- 返回4条不同路径

**注意**: 路径3和4的里程异常高（存在环路），但函数正常工作，能够找到多条替代路径。

## 第九阶段：intervalgroup 修复模块开发（2026-04-16 ~ 2026-04-17）

### 背景与目标

**用户请求**:
> 连接 Hive 数据库（172.16.5.1:10000, db: dbbase2026），修复表 `gstx_exit_with_min_fee202603` 中的 `intervalgroup` 字段。

**问题描述**:
该字段记录车辆通行经过的收费单元ID序列（用"|"分隔），存在两类问题：
1. **遗漏**: 相邻单元在拓扑上不相邻，中间有缺失
2. **错误**: 中间单元方向识别错误（应为反向单元）

**涉及数据量**: 约4880万条记录

---

### 技术方案设计

#### 滑动窗口算法

对每个 intervalgroup 序列，使用滑动窗口处理四种情况：

| 情况 | X1→X2 | X2→X3 | 处理逻辑 |
|------|-------|-------|----------|
| 1 | ✅ | ✅ | 窗口滑动 |
| 2 | ❌ | ✅ | 补充最短路径 |
| 3 | ❌ | ❌ | 尝试反向，取最短路径 |
| 4 | ✅ | ❌ | 窗口滑动 |

**关键函数**:
- `topo_next(section_id)` → 拓扑后继集合（基于 `dwd_tom_noderelation` 表）
- `topo_check(from, to)` → 检查是否相邻
- `shortest_path(start, end)` → pgRouting 最短路径（基于 `find_shortest_path_pgr` 函数）
- `reverse_section_id(X2)` → 计算反向单元ID（末2位 10↔20 互换）

#### 去重逻辑

当补充路径的终点与原始序列中的元素重复时：
- `chosen[-1] == X3`: 删除原始X3，终点已在chosen中
- `chosen[-1] == X2`: 删除原始X2，终点已在chosen中
- 其他: 正常添加终点

### 遇到的问题及修复

#### 问题1：Hive 连接 - 密码特殊字符

**现象**:
- Hive 密码包含 "@" 字符，导致连接 URL 解析失败

**解决方案**:
- 使用 `urllib.parse.quote` 对密码进行 URL 编码
```python
from urllib.parse import quote
password = quote(self.password, safe="")
```

#### 问题2：Hive 子查询内存溢出

**现象**:
- 初始使用子查询方式读取数据，导致内存溢出

**解决方案**:
- 改为直接 SELECT 查询，不使用子查询

#### 问题3：shortest_path() 返回格式问题

**现象**:
- 路径查询函数返回格式不正确

**解决方案**:
- 改为访问 `row[1]`（node_path 数组字段）而非遍历行

#### 问题4：重复单元出现

**现象**:
- 修复结果中出现重复的收费单元

**解决方案**:
- 添加去重检查：`if node not in result_list`

#### 问题5：最后一个元素丢失（重要Bug）

**现象**:
- 序列 `110|210|310|120|220` 修复后，120单元丢失

**问题分析**:
- `topo_check(310, 120) = False`
- `topo_check(120, 220) = False`
- 进入**情况3（2b分支）**：最短路径终点是120，与原始X2相同
- 代码使用 `chosen[1:-1]` 去掉了首尾，导致终点（X2=120）丢失

**原错误代码**:
```python
# 添加 chosen 的中间节点（去掉首尾，因为首=X1，尾=终点）
for node in chosen[1:-1]:  # ❌ 错误：去掉了终点
    if node not in result_list:
        result_list.append(node)
```

**修复后代码**:
```python
# 添加路径节点（去掉起点X1，保留终点）
for node in chosen[1:]:  # ✅ 正确：保留终点
    if node not in result_list:
        result_list.append(node)
```

修复: G000561001000110|G000561001000210|G000561001000310|G000561001001520|G0005610010030|G000561001001510|G000561001000320|G000561001000220|G000561001000120

- 数据源从CSV文件读取（不再从Hive），约4880万条、>10GB

- 每个收费单元的小时归属 = 该单元在修复后 `intervaltimegroup` 中对应时间截断到小时（带日期，如 `2026-03-01 12:00:00`）
- 去重规则：同一记录内，同一 (section_id, stat_hour) 只算1次流量
- `od_section_path_id` 在map表中查不到时，直接INSERT一条新记录

---

### 核心设计决策

#### 1. intervaltimegroup 同步修复策略

**选项**：同步构建（改内部） vs 事后对齐插值

**决策**：同步构建 — 在 `fix_intervalgroup()` 内部，构建 `result_list` 的同时构建 `time_list`

| 情况 | section处理 | time_list处理 |
|------|------------|--------------|
| 1: X1→X2✅, X2→X3✅ | X2加入result_list | seq_times[i+1] 加入time_list |
| 2: X1→X2❌, X2→X3✅ | shortest_path插入中间节点 | 两端已知时间之间等间隔插值 |
| 3a: reverse_fix | X2→X2_rev | 沿用 seq_times[i+1]（被替换节点的原始时间） |
| 3b: path_fill/reverse_path_fill | shortest_path插入+可能删除seq节点 | 插值；删除seq节点时同步删除seq_times |
| 4: X1→X2✅, X2→X3❌ | X2加入result_list | seq_times[i+1] 加入time_list |

#### 2. 重复section时间处理

**决策**：保留首次出现时间（`if node not in result_list` 跳过时，时间也同步跳过）

#### 3. od_section_path_id 查找/插入

**决策**：查不到就INSERT — 使用 `INSERT ... ON CONFLICT DO UPDATE SET updated_at = CURRENT_TIMESTAMP RETURNING id`

### 失败处理机制（shortest_path 查询失败）

**问题**：`intervalgroup` 补全过程中调用 `shortest_path()` 时，可能因路网数据缺失导致查询失败（返回 `None`），若直接抛异常会中断整个批处理流程。

**解决方案**：

1. **`IntervalFixResult.error` 字段** — 标记失败原因
   - Case 2（path_fill）：`error = "path_fill_failed:X1->X2"`
   - Case 3b（reverse_path_fill）：`error = "path_fill_failed:X1->X2(case3b)"`

2. **`FixFailureLogger` 模块** — 独立记录失败详情到本地CSV
   - 文件位置：`{output_dir}/fix_failures/fix_failures_v{version}.csv`
   - 记录字段：`enid, exid, intervalgroup, intervaltimegroup, envehicleid, exvehicleid, entime, extime, failure_reason`
   - 追加模式：支持多次运行自动追加，不重复写header
   - 按版本分文件：不同版本数据写入不同CSV

3. **Service层跳过逻辑** — `_process_batch()` 中检测到 `fix_result.error` 时：
   - 调用 `FixFailureLogger.log_failure(record, fix_result.error)` 记录
   - `self._fix_failures += 1` 计数
   - `continue` 跳过该记录，继续处理下一条

4. **`FlowStatResult` 扩展** — 新增 `fix_failures: int` 字段，最终报告中可查看修复失败记录数

### 遇到的问题及修复

#### 问题1：shortest_path mock 传参格式错误

**现象**：测试中 `paths.get` 作为 `side_effect` 时，`(a, b)` 被解析为 `dict.get(key, default)`，导致路径查找永远返回 None

**解决方案**：改为 `lambda a, b: paths.get((a, b))`

#### 问题2：reverse_section_id 测试使用非标准ID格式

**现象**：`"B_rev"` 末2位不是 10/20，`reverse_section_id` 返回 None，无法触发 reverse_fix

**解决方案**：使用标准格式的section ID（如 `G007061003000220` → `G007061003000210`）

#### 问题3：_process_batch 测试中 _flow_agg 类型不匹配

**现象**：服务代码使用 `defaultdict(int)`，但测试中初始化为普通 `dict`，导致 `KeyError`

**解决方案**：测试中也使用 `defaultdict(int)`

#### 问题4：interpolated_times 插值数量与测试预期不符

**现象**：`shortest_path("A","B")` 返回 `["A","X","B"]`，即新增2个节点，测试预期只插入1个时间（10:10:00）

**解决方案**：2个新增节点在 10:00~10:20 之间等间隔插值，实际为 10:06:40 和 10:13:20，修正断言

#### 问题5：timegroup 长度短于 section 数量时的空字符串处理

**现象**：`intervaltimegroup` 字段比 `intervalgroup` 短时，缺失时间用空字符串填充，`split_intervalgroup` 会过滤掉空字符串，导致 times 长度可能少于 sections

**解决方案**：测试断言改为 `len(times) >= 2`，验证不崩溃且已有时间被保留

## 第十二阶段：M2 多进程并行化改造（2026-04-28）

### 问题背景

第十一阶段完成的 M2 小时流量统计模块以**单进程**运行，处理一个月 3160 万行 CSV 数据需要约 **10 天**。核心瓶颈：

1. `fix_intervalgroup_batch()` 纯串行 CPU 密集计算
2. `upsert_od_path_map()` 逐行 DB 往返（缓存未命中时）
3. `_flow_agg` 全局聚合字典在 flush 间隔期间无限增长（5 个 batch × 50 万行 = 250 万行聚合量）
4. `iter_csv_batches` 整批 50 万行 list[dict] 驻留内存

目标：**10 个进程并行，运行时间从 10 天压缩到 1 天**，同时控制内存。

### 架构设计：分区并行 + 即时刷盘

```
                    Main Process
                        │
        ┌───────────────┼───────────────┐
        │               │               │
   CSV 预扫描       启动 Worker Pool    结果汇总
   (行偏移索引)     (multiprocessing.Pool)
        │               │               │
        ▼               ▼               ▼
  ┌─────────┐   ┌───┐ ┌───┐ ┌───┐   ┌──────────┐
  │ 偏移索引  │   │ W0│ │ W1│ │ W9│   │ 验证+摘要 │
  └─────────┘   └───┘ └───┘ └───┘   └──────────┘
                  │      │      │
                  ▼      ▼      ▼
              各 Worker 独立读取 CSV 分区
              各 Worker 独立 DB Session
              各 Worker 即时 flush → ON CONFLICT 累加
```

**核心思路**：
- **分区读取**：预扫描 CSV 建立行偏移索引，每个 Worker seek 到指定位置读取，避免预拆分文件
- **即时刷盘**：每个 mini-batch 处理完立即 flush 到 DB，消灭全局 `_flow_agg` 字典
- **ON CONFLICT 累加**：`flow_cnt = table.flow_cnt + EXCLUDED.flow_cnt` 保证多进程写入正确性
- **小批量流式**：batch_size 从 50 万降到 5 万，每批处理后立即释放内存

### 核心设计决策

#### 1. CSV 分区策略：行偏移索引

**不预拆分文件**，而是预扫描一次 CSV，记录每 N 行的 byte offset。Worker i 被分配指定偏移范围，直接 seek 到该位置读取。HDD 上 seek 到分区开头只需一次，之后是顺序读取。

#### 2. 内存控制：mini-batch + 即时 flush

| 参数 | 改造前 | 改造后 | 说明 |
|------|--------|--------|------|
| batch_size | 500,000 | 50,000 | 每次 CSV 读取量 |
| upsert_interval | 5 | 1 | 每 1 个 mini-batch 就 flush |
| 全局 _flow_agg | 250 万行聚合 | 无 | 每个 mini-batch 后立即清空 |
| 单 Worker 内存 | ~400MB | ~40MB | 5万行list + 小聚合dict |
| 10 Worker 总内存 | N/A | ~400MB+共享~50MB | 远低于 48GB 上限 |

#### 3. DB 并发安全 — ON CONFLICT 行级原子累加

PostgreSQL 的 `INSERT ... ON CONFLICT DO UPDATE` 是行级原子操作：
- 两个 Worker 写不同行 → 完全并行，互不阻塞
- 两个 Worker 写同一行 → 第二个等待行锁释放后累加 `flow_cnt`

每个 Worker 拥有独立 DB Session，连接池 10+20 overflow = 30 连接，10 Worker × 1 连接 = 10，安全。

#### 4. fork 继承优化（Linux 生产环境）

主进程在创建 Pool 前预加载所有共享数据（section_map, od_path_lookup, topology_cache），fork 后子进程通过 CoW 共享，零拷贝开销。Windows spawn 兼容：Worker 函数内部检测共享数据是否存在，不存在则自行加载。

#### 5. TopologyChecker 多进程适配

`_pg_connection` 延迟到首次 DB 调用时创建，fork 后调用 `_reset_pg_connection()` 重置连接，确保子进程建立独立连接。`_next_cache` 通过 fork CoW 共享，只读访问不触发拷贝。

#### 6. 批量化 upsert_od_path_map

原逐行 INSERT...RETURNING 改为：每个 mini-batch 收集所有缓存未命中 → 一次性批量 INSERT...ON CONFLICT...RETURNING → 返回 ID 更新本地缓存。减少约 90% 的 DB 往返次数。

## 第十三阶段：M2 日文件处理改造（2026-04-29）

### 背景与目标

**用户请求**:
> 月文件 `gstx_exit_with_min_fee202603.csv` 太大（>10GB），已拆分为日文件放在 `/home/shy/gaosu_data/202603/data_20260301.csv` ~ `data_20260331.csv`（每文件约 1.2M 行/450-650MB）。需要改造 M2 支持按月份指定目录，自动发现日文件，并改为日表输出。

**核心需求**:
1. 按月份指定目录，自动发现日文件
2. 日文件 Round-Robin 分配给 Worker 并行处理
3. **输出改为日表**：`dws_section_od_path_flow_hour_{YYYYMMDD}`，每天一张表

---

### 核心设计决策

#### 1. 双版本机制

| 操作 | 使用的版本 | 说明 |
|------|-----------|------|
| `dwd_od_section_path_map` 查找/插入 | 月版 `{YYYYMM}`（如 `202603`） | OD 路径映射按月管理 |
| `dws_section_od_path_flow_hour_{version}` 建表/upsert | 日版 `{YYYYMMDD}`（如 `20260301`） | 流量统计按天输出 |

#### 2. 三路分发逻辑

```
--data-dir 非空？
  ├─ 是 → 日文件模式
  │      ├─ --workers 1 → _run_sequential_daily()
  │      └─ --workers N → _run_parallel_daily()（Round-Robin 分配日文件）
  └─ 否 → 月文件模式（保持不变）
         ├─ --workers 1 → _run_sequential()
         └─ --workers N → _run_parallel()（字节偏移分区）
```

#### 3. 日文件 has_header 自动检测

`split_by_month.py` 早期版本生成的日文件不含表头，日文件模式自动调用 `_detect_has_header()` 检测首行是否包含 `enid`/`exid`/`intervalgroup` 等关键字。同时修复 `split_by_month.py` 补写表头。

#### 4. DDL 并发安全

并行模式下，所有日表的 `create_table(day_version)` 在主进程中执行，避免多 Worker 同时 DDL 冲突。

### 遇到的问题及修复

#### 问题1：日文件无表头导致首行丢失

**现象**：`split_by_month.py` 只写数据行不写表头，`iter_csv_batches()` 默认 `next(csv.reader(f))` 跳过首行导致每个日文件丢失第一条记录。

**解决方案**：
1. 修复 `split_by_month.py` 写入表头（根因修复）
2. 新增 `_detect_has_header()` 自动检测（兼容旧文件兜底）
3. `iter_csv_batches()` 新增 `has_header` 参数

#### 问题2：CSV 列顺序不匹配

**现象**：月文件表头为 `exid,enid,...`（exid 在前），但 `CSV_COLUMNS = ["enid","exid",...]`。当 `has_header=False` 时按位置映射，导致 enid/exid 反转。

**解决方案**：新增 `CSV_COLUMNS_IN_FILE_ORDER = ["exid","enid",...]`，`has_header=False` 时使用此常量。

#### 问题3：`_detect_has_header` 对空文件返回 False

**现象**：空文件 → `f.readline().strip().lower()` 返回 `""` → `any(keyword in "" for ...)` 返回 `False`。

**解决方案**：添加 `if not first_line: return True` 检查（安全降级）。

#### 问题4：`build_csv_offset_index` 返回元组导致旧测试失败

**现象**：函数返回 `(offsets, line_count)` 元组，旧测试直接比较 `> 0` 导致 `TypeError`。

**解决方案**：测试中改为元组解包 `offsets, line_count = build_csv_offset_index(...)`。

#### 问题5：fork 后 `engine.dispose()` SSL 错误

**现象**：子进程继承父进程 DB 连接池，`dispose()` 尝试 rollback 失效连接时报 `SSL error: unexpected eof while reading`。

**解决方案**：`dispose()` 外包裹 `try/except` 捕获异常直接忽略（fork 后旧连接应直接丢弃）。

## 第十四阶段：M3 交通影响分析 — 受影响OD查询与中途下站检测

### 阶段概述

实现 M3 交通影响分析的两个核心流程：
- **流程1**：施工收费单元 → 查找受影响OD → 查询所有Path流量 → 输出CSV
- **流程2**：基于受影响OD对，从原始CSV中检测中途下站车辆
- **串联入口**：一键执行流程1 → 流程2

### 流程1：受影响OD-Path流量查询

#### 核心逻辑

1. 输入施工收费单元ID（`sectionIds`，`|` 分隔）和施工时间段
2. 查找适用版本的 `dwd_od_section_path_map`
3. 查找受影响OD-Path（path 经过施工段）
4. 提取受影响OD对集合 → 查询这些OD下**所有**path（含受影响和不受影响）
5. 标记 `is_affected`、`affected_section_ids`
6. 多版本去重（同一 enid+exid+numpath 取最新版本，同版本合并 affected_section_ids）
7. 查询施工期间和2025同期流量（优先日表，回退月表）
8. 流量过滤（`minFlow` 参数）
9. OD对过滤（只保留至少有一条 `is_affected=True` path的OD）
10. 输出CSV

#### 关键修复：Cartesian Product 去除

**问题**：`find_all_paths_for_ods` 原实现使用 `enid=ANY(:enid_list) AND exid=ANY(:exid_list)`，当 OD 对为 `(A→B, C→D)` 时，实际查出了 `A→B, A→D, C→B, C→D` 四种组合（Cartesian 乘积），导致 209,959 行（预期 ~10,000 行）。

**修复**：改用 `VALUES CTE + JOIN` 精确匹配 OD 对：

```python
values_clauses = ", ".join(f"('{enid}', '{exid}')" for enid, exid in od_pairs)
sql = f"""
    WITH target_ods(enid, exid) AS (VALUES {values_clauses})
    SELECT m.id, m.enid, m.exid, m.numpath, m.fixed_intervalpath, m.version_yyyyMM
    FROM dwd_od_section_path_map m
    JOIN target_ods t ON m.enid = t.enid AND m.exid = t.exid
    WHERE m.version_yyyyMM = ANY(:versions)
"""
```

修复后查询返回 9,913 行，耗时从 71s 降至 21s。

#### 关键修复：SQLAlchemy `::varchar[]` 语法问题

**问题**：`get_applicable_versions` 原实现使用 `unnest(:months::varchar[])` 语法，SQLAlchemy `text()` 将 `::` 解释为绑定参数前缀。

**修复**：改为 Python 端过滤 — 先查全部版本，再用集合交集筛选适用版本。

#### 关键修复：数据库列名大小写

**问题**：PostgreSQL 返回列名统一为小写，`version_yyyyMM` 实际为 `version_yyyymm`。

**修复**：所有引用该列的代码统一改为 `version_yyyymm`。

#### 流量过滤（Step 10）

`minFlow` 参数：只保留 `construction_flow > N OR same_period_2025_flow > N` 的 path。

```python
if params.minFlow > 0:
    records = [
        r for r in records
        if (r.construction_flow is not None and r.construction_flow > params.minFlow)
        or (r.same_period_2025_flow is not None and r.same_period_2025_flow > params.minFlow)
    ]
```

#### OD对过滤（Step 11）

过滤后只保留至少有一条 `is_affected=True` path 的 OD 对下的所有 path，同时提取 `filteredOdPairs` 供流程2使用。

```python
od_has_affected = {(r.enid, r.exid) for r in records if r.is_affected}
if od_has_affected:
    records = [r for r in records if (r.enid, r.exid) in od_has_affected]

```

### 1. 流程1：日表流量累加Bug修复

**问题**：施工期间流量统计结果异常偏低，经排查发现日表查询时 `flow_results.update(day_flows)` 仅保留最后一天数据，且 `missing_path_ids` 逻辑导致第2天起跳过已有path_id。

**根因**：
- `flow_results.update(day_flows)` 是覆盖式合并，day N 的结果会覆盖 day 1~N-1 的同名 key
- `missing_path_ids = [pid for pid in path_ids if pid not in flow_results]` 导致 day 1 已填充的 path_id 在 day 2+ 被跳过

**修复**：
```python
# 修复前（Bug）
flow_results.update(day_flows)

# 修复后（累加）
for pid, flow in day_flows.items():
    flow_results[pid] = flow_results.get(pid, 0) + flow
```

同步修复了2025同期日表查询和月表回退路径中的同类问题。

### 第30步：流程3绕行流量均分逻辑

**背景问题**:
> 流程3中，一条CSV记录 `(rec_enid, rec_exid)` 可能同时匹配多个受影响OD对 `(od_enid, od_exid)`，每匹配一个OD对就 +1 计入流量，导致同一物理行程被重复统计，流量虚高。

**解决方案**:
在流量统计汇总前，对同一 `(rec_enid, rec_exid, record_type, vehicle_type)` 映射到 N 个不同 `(od_enid, od_exid)` 的情况，将流量和费用均除以 N。

**关键设计决策**:
- 除数基于两期（施工期 + 2025同期）合并计算——同一物理行程的OD映射关系不因期间不同而变
- 均分仅作用于汇总统计，明细CSV（DetourRecordRecord）保持不变
- `DetourFlowStatRecord` 的 `construction_flow` 和 `same_period_2025_flow` 从 `int` 改为 `float`

**修改文件**:

|------|---------|

**新增函数** `_average_flow_by_od_count()`:

## 第十八阶段：M7 数据挖掘模块开发（2026-05-12）

### 背景与目标

**用户需求**:
新增数据分析模块，包含两个功能：
1. **流失高频车辆挖掘** — 根据OD列表和时间段，从日表CSV中筛选匹配OD的通行记录，统计车牌出现频次
2. **绕行高频路段挖掘** — 根据OD列表（含流量X）和基础表，通过累加construction_flow找出绕行高频路段

模块编号：M7，作为正式模块纳入 `src/modules/` 体系。

---

### 核心设计决策

#### 1. OD输入格式自动识别

- 长度>9 → enid/exid 格式，直接匹配日表的 enid/exid 字段
- 长度≤9 → section_number 格式，需加载 PG 映射后转换
- 支持混合输入

#### 2. 双向匹配

- 任意方向满足都算：OD(A,B) 同时匹配 (A→B) 和 (B→A)

#### 3. 车牌+车型联合计数

- vehicle_id 优先取 exvehicleid，为空取 envehicleid
- vehicle_type 取 feevehicletype 字段
- key 为 `(vehicle_id, vehicle_type)` 元组

#### 4. 绕行路段流量累加逻辑

每个OD处理：
1. 从基础表筛选 `is_affected=False` 的记录
2. 按 numpath 分组，汇总所有 vehicle_type 的 construction_flow
3. 按 construction_flow 升序排序，从小到大累加
4. 累加值超过 flow_x 时，最后一个 numpath 取部分量
5. 拆分 numpath 为 section_number 列表，累加各 section 的流量

## 第二十阶段：动态测算最后一公里实现（2026-05-20）

### 背景

动态测算主流程 `run_affected_od_and_mid_trip.py` 已完成流程1（受影响OD流量查询）、流程2（中途下站检测）、流程3（绕行记录检测），但综合汇总模块使用英文列名和"均值×流量"估算模型，无法直接支撑"动态测算"业务需求。需要重构为7步流程，输出中文列名，与现有 notebook 分析逻辑对齐。

### 第20步：重构 ImpactSummaryService 为7步流水线

**用户请求**:
> 动态测算最后一公里实现：将流程1数据按is_affected透视、流程2/3数据聚合重命名、Left Join关联、派生计算（受影响流量、绕行流量、保留路径、流失流量、通行费等）、OD/路段汇总表、流失分析、高频车辆查询。

**实现过程**:

#### 步骤1：Flow1 聚合 + is_affected 透视

- 分组键: `(enid, exid, vehicle_type, section_od, is_affected)`
- 对 `fee_yuan`, `control_fee_yuan`, `construction_flow`, `same_period_2025_flow`, `total_length_meters`, `control_length_meters` 求和
- 按 `is_affected` 透视合并为单行，字段重命名为中文（施工影响路径/其他路径前缀）
- 过滤: `施工影响路径2025年同期流量 <= 0` 的行删除

#### 步骤2：Flow2/Flow3 聚合 + 重命名

- Flow2: 按 `(od_enid, od_exid, vehicle_type)` 求和，丢弃 sp2025 字段，重命名为"中途上下站流量"等
- Flow3: 同样聚合，重命名为"附近上下车流量"等
- 新增"单次通行损失费额"和"单次通行交控损失费额"（损失费/流量）
- 流量为0时单次费率设为0（避免除零）

#### 步骤3：Left Join

- 步骤1结果 LEFT JOIN 步骤2/3结果，关联键 `(enid, exid, vehicle_type)`
- 未匹配: 生成字段名，值设为0

#### 步骤4：派生计算（核心业务逻辑）

- `受影响流量 = MAX(0, 参考总流量 - 未收影响流量)`
- `绕行流量 = MIN(MAX(0, 其他路径施工期间流量 - 其他路径2025年同期流量), 受影响流量)`
- `保留路径(附近上下站)`: 嵌套IF逻辑，基于附近上下车增量、中途上下站增量、受影响流量余额
- `保留路径(中途上下站)`: 类似嵌套IF
- `流失流量 = 受影响流量 - 绕行流量 - 保留路径(附近上下站) - 保留路径(中途上下站)`
- 所有通行费除以10000转万元
- 提取为独立纯函数 `_compute_retained_nearby`/`_compute_retained_midtrip`，便于单元测试

#### 步骤5：OD汇总表 + 路段汇总表

- OD汇总表: 按 `(enid, exid)` 汇总13个数值字段
- 路段汇总表: 按 `section_od` (OD_num) 汇总同13个字段

#### 核心算法 V1（相邻补全方案）

```
1. 解析 raw_path → 节点列表
2. 连续重复去重
3. 过滤非法节点（拓扑中不存在的）
4. 遍历相邻节点对 (A, B)：
   a. 如果 A→B 有直接拓扑边 → 保留
   b. 如果 A→B 不可达但 shortest_path(A, B) 存在 → 用最短路径补全
   c. 如果 A→B 不可达 → 在后续 max_gap_search_window 个节点中找第一个可达的
   d. 如果都不可达 → 标记 LOW_CONFIDENCE
5. 匹配经纬度
6. 计算基础指标（重复节点、绕行比、折返指数）
7. 输出修正结果
```

### Phase 2：V2 锚点 DP + 分段 KSP

#### V2 算法流程

```
1. 路径标准化（同 V1）
2. 候选锚点筛选：detour_ratio = (d_start_v + d_v_end) / d_start_end
3. DP 选择最优锚点链（分段成本 + 跳过惩罚 + 绕行惩罚）
4. 分段 K 最短路拼接：
   a. 对每对相邻锚点查询 K 条候选路径（pgRouting pgr_ksp）
   b. LCS 观测匹配评分 + 插入/跳过/折返惩罚
   c. 选择评分最优候选拼接
5. 后处理（相邻去重、小环识别、允许一次掉头）
6. 若 V2 不可行则回退 V1
```

## 第二十二阶段：M8 路径修正功能增强 — 最短路径、施工路段检查与绕行路径（2026-05-25）

### 背景与目标

**用户请求**:
在 M8 路径修正功能中新增以下输出：
1. 最短路径（可能经过施工路段）
2. 经过施工路段的最短距离
3. 绕行路径（不经过施工路段的最短路径）
4. 最小费额路径

**设计原则**:
- 绕行路径只在 K 条最短路径有备选方案时才输出
- 最小费额路径只针对绕行路径计算，不针对最短路径
- 最小费额路径通过 K 条最短路径 + 费额计算得到

### 核心实现

#### 1. pipeline.py 修改（Step 9）

```python

shortest_path_cost = graph.shortest_path_cost(start_node, end_node)
k_paths = graph.k_shortest_paths(start_node, end_node, k=5)

# 施工路段检查
passes_construction = None
shortest_through_construction = None
detour_path = None
detour_distance = None
detour_fee = None
min_fee_path = None
min_fee = None

if construction_sections and shortest_path_nodes:
    sp_set = set(shortest_path_nodes)
    const_set = set(construction_sections)
    passes_construction = bool(sp_set & const_set)

    if passes_construction:
        shortest_through_construction = shortest_path_cost

        if k_paths:
            # 从K条最短路径中找不经过施工路段的路径
            detour_paths = []  # (path, cost, fee)
            for path, path_cost in k_paths:
                path_set = set(path)
                if not (path_set & const_set):
                    fee_result = toll_calc.calculate_path_fee(path, vehicle_type, graph.version)
                    fee = fee_result.fee_yuan if fee_result else None
                    detour_paths.append((path, path_cost, fee))

            if detour_paths:
                # 最短绕行路径（距离最短）
                detour_path, detour_distance, _ = min(detour_paths, key=lambda x: x[1])

                # 最小费额绕行路径
                valid_paths = [(p, c, f) for p, c, f in detour_paths if f is not None]
                if valid_paths:
                    min_fee_path, min_fee_cost, min_fee = min(valid_paths, key=lambda x: x[2])
```
